Executive Summary

The codebase has a solid foundation: Supabase RLS is taken seriously, the enriched events RPC avoids the worst N+1 pattern on the primary event feed, React Query is in place, Sentry is integrated across browser and edge functions, and unit coverage exists for several domain utilities. The main production risks are concentrated in release gating, security hygiene, bundle size, data-access discipline, and observability defaults.

The most urgent issues are a tracked credential-looking file, a broken local quality gate, unsafe Sentry defaults, no CI release gate, no route-level code splitting, admin queries that load full tables, and client-side filtering after limited event fetches. The current architecture is workable for an MVP, but the layer-first folder layout and broad query invalidation will become expensive as admin, ingest, discovery, and personalization grow.

Make Sentry production-safe by default

P0 Observability Security Privacy
Problem

src/lib/sentry.ts enables Sentry with a hard-coded fallback DSN, tracesSampleRate: 1.0, sendDefaultPii: true, console log capture, and replay integration defaults. User context is not set from the auth provider, so the app sends more ambient data than needed while missing explicit app-level identity metadata.

Impact

Local/dev traffic and unintended environments can pollute production telemetry. Full tracing and default PII are expensive and risky. Missing explicit user context makes triage slower and encourages over-collection elsewhere.

Recommendation

Require VITE_SENTRY_DSN in production, disable Sentry entirely when absent, set a low sample rate by environment, disable default PII, and explicitly set redacted user context after auth. 

const dsn = env.VITE_SENTRY_DSN

if (dsn && import.meta.env.PROD) {
  Sentry.init({
    dsn,
    environment: env.VITE_SENTRY_ENVIRONMENT ?? import.meta.env.MODE,
    release: env.VITE_SENTRY_RELEASE,
    sendDefaultPii: false,
    tracesSampleRate: Number(env.VITE_SENTRY_TRACES_SAMPLE_RATE ?? 0.05),
    replaysSessionSampleRate: 0,
    replaysOnErrorSampleRate: 0.1,
  })
}

Push event search, filters, and pagination into Supabase

P1 Supabase Performance Scalability
Problem

Explore fetches enriched events for a city, then applies keyword, age, tag, date, free-only, and distance filtering in React. The RPC defaults to a finite page size, so client-side filtering can produce false empty states when matching events exist outside the first page. The legacy search_events RPC uses ILIKE instead of the existing search_vector index.

Impact

Search becomes slower and less correct as event volume grows. More data crosses the network than needed, caching is less effective, and pagination cannot be reasoned about accurately.

Recommendation

Create one server-side discovery RPC that accepts normalized filter parameters, uses full-text search or trigram indexes for keyword search, applies tag and date filters in SQL, and returns a cursor or total-aware page result. Keep client-only filters only for presentation details.

  p_date_to => $5,
  p_limit => 24,
  p_cursor => $6

Replace broad query invalidation with a query-key contract

P1 State Management React Query Performance
Problem

Favorite, calendar, rating, and admin mutations invalidate multiple broad key prefixes such as ["events"], ["events-enriched"], ["event", id], and ["events-by-id"]. Query key shapes are manually repeated across hooks.

Impact

Mutations trigger avoidable refetches and make cache behavior hard to predict. As pages multiply, stale UI bugs and unnecessary network traffic will grow.

Recommendation

Add a centralized query key factory, use stable normalized params, and update known cache entries directly for simple toggles. Reserve broad invalidation for true cross-cutting changes such as publishing or deleting an event. 

export const qk = {
  events: {
    enriched: (params: EnrichedEventParams) => ["events", "enriched", normalize(params)] as const,
    detail: (id: string, userId?: string) => ["events", "detail", id, userId ?? null] as const,
  },
  user: {
    favorites: (userId: string) => ["user", userId, "favorites"] as const,
  },
}

Harden edge ingestion against SSRF and resource exhaustion

P1 Security Supabase Edge Functions Reliability
Problem

Edge functions validate URL literals and private IP patterns, but the code documents that DNS rebinding is not covered. Fetched response bodies are read as full text without a strict byte cap. CORS helpers allow * and broad methods even for service-role/admin flows.

Impact

Admin-submitted source URLs can become an SSRF or resource exhaustion path. Overbroad CORS and methods increase the attack surface, even when authorization checks are otherwise strong.

Recommendation

Resolve DNS and reject private resolved addresses before fetch, enforce content-length and streamed byte caps, restrict production origins and methods, and add regression tests for localhost, private network, redirect, oversized, and rebinding cases. 

const maxBytes = 2_000_000
const length = Number(response.headers.get("content-length") ?? 0)
if (length > maxBytes) {
  throw new Error("Source response too large")

P1 Reliability Supabase Data Quality
Problem

Impact

  • Context churn: auth and app context values should be memoized or split so small state changes do not rerender unrelated consumers.
  • Effect handling: auth bootstrap and invite claim paths silently catch failures. Capture and expose retryable errors.

Zustand Review

Zustand is not present. That is not inherently a problem. React Query is the correct owner for server state, and most current global state can remain in context if memoized and kept small.

If Zustand is introduced, the production-grade boundary is UI state only: selected city, map viewport, sidebar state, temporary filter drafts, or local preferences. Do not store events, profile rows, access grants, favorites, ratings, or Supabase records in Zustand. Those belong in React Query and Supabase.

Supabase Review

  • Good: RLS policy work, force-RLS migrations, security-invoker view usage, and the enriched events RPC show strong security intent.
  • Query efficiency: replace .select("*"), full-table admin fetches, and client-side Explore filtering.
  • Search: move keyword search to indexed full-text or trigram search and stop relying on ILIKE '%term%'.
  • Pagination: make pagination explicit for public discovery and admin lists.
  • Realtime: define where realtime is required. Most surfaces can use stale-time and targeted invalidation; admin source runs may justify polling or channel subscriptions.
  • Security: add advisor/RLS checks in CI and reduce edge CORS/method scope.
  • Edge functions: harden SSRF, response-size limits, dedupe error handling, external-call timeouts, and run idempotency.

Sentry Review

  • Coverage: browser, React Query cache, and edge functions are instrumented. That is a good baseline.
  • Defaults: hard-coded DSN, full tracing, default PII, and console capture are too aggressive.
  • User context: set redacted user id, role/access level, and city context after auth; clear on signout.
  • Tracing: add route names, query mutation names, and edge-function spans for scrape/import phases.
  • Noise: filter expected auth, validation, aborted fetch, and user-cancelled errors before sending.
  • Source maps: upload is wired through Vite config. Make release/env required in CI so source maps match deployed artifacts.

TailwindCSS Review

  • Token usage: the app has CSS variables and Tailwind 4 tokens, but product code still uses raw utility colors.
  • Duplication: filter chips, badges, admin status labels, and action buttons should use shared variants.
  • Responsive behavior: verify long event names, locations, and admin row actions on mobile widths.
  • Dark mode: token definitions exist, but raw color utilities will not automatically adapt cleanly.
  • Palette: the app leans heavily into a single warm family. Keep the brand, but expand semantic status colors through tokens rather than raw color utilities.

Potential Overengineering

  • Zustand migration: unnecessary unless the team defines a small UI-only boundary. React Query already solves server cache.
  • Realtime everywhere: most event discovery surfaces do not need realtime. Use it only for source-run progress or collaborative/admin flows.
  • Feature folder rewrite: valuable, but should be incremental. Start with admin and events instead of moving the whole repo in one PR.
  • Custom design system: avoid building a large abstraction layer. A few variants for repeated primitives will buy most of the consistency.
  • Premature normalization: normalize client state only where cached relationships are actually mutated. Do not reimplement a client database.

Things Done Well

  • React Query is already the server-state foundation, which avoids many common custom-cache mistakes.
  • The enriched events RPC is a strong direction and has targeted tests around key event enrichment behavior.
  • Supabase RLS and security-invoker view patterns are treated as first-class concerns.
  • Sentry source-map upload and edge-function error capture exist, even though defaults need tightening.
  • The test suite is not empty: 149 unit tests passed during audit.
  • The dependency set is current. Only react-day-picker reported a newer major version.
  • Tailwind tokens and UI primitives provide a foundation for a more consistent design system.

Modernization Opportunities

  • Consider TanStack Router if route-level data loading, route error boundaries, and typed URL search params become more important than the current simple React Router setup.
  • Introduce generated Supabase type drift checks and typed RPC wrappers before adding more database functions.
  • Add bundle analysis to CI so route-splitting progress is measured and regressions are visible.
  • Add OpenTelemetry-style span naming consistency across Sentry browser and edge instrumentation.
  • Use database-backed cursor pagination for discovery instead of offset pagination if event volume grows quickly.
  • Add a small UI-state store only after context memoization and query-key cleanup are complete.
